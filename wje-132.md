## 项目主要需求

### 项目概述

【全栈Web应用】建筑施工安全管理平台。面向工地安全管理员和监理人员，支持安全隐患排查、事故记录与跟踪、安全培训管理和人员资质证书管理。**本项目用于验证模型跨文件协同改动能力，禁止将多个职责合并进单一文件。**

### 核心实体（≥ 3 个，每个贯穿全栈）

1. **安全事件（SafetyIncident）** — 字段：事件标题、描述、发生时间、发生地点（工地+区域）、严重等级（NearMiss/Minor/Moderate/Major/Fatal）、分类（坠落/触电/物体打击/坍塌/机械伤害/其他）、涉及人员ID列表、现场照片URL列表、处理状态（Reported/Investigating/Resolved/Closed）、整改措施、整改截止日期、上报人ID
   - 业务动作：上报事件、指派调查、提交整改、关闭事件
   - **贯穿全栈**

2. **安全检查（SafetyInspection）** — 字段：检查名称、检查类型（Routine/Special/PreShift/Emergency）、检查区域、检查日期、检查人ID、总评分（0-100）、检查状态（Scheduled/InProgress/Completed/Failed）、发现的隐患数量、合格项数量
   - 关联：1次检查 → 多个检查项（InspectionItem：检查项名称、是否合格、备注、照片URL）
   - 业务动作：创建检查计划、执行检查、记录结果、生成报告
   - **贯穿全栈**

3. **安全培训（SafetyTraining）** — 字段：培训主题、培训类型（Induction/Regular/Special/Emergency）、培训日期、培训时长（小时）、培训讲师、培训地点、培训内容摘要、参训人员ID列表、考核方式（笔试/实操/口试）、通过率
   - 业务动作：创建培训、记录签到、录入成绩、生成培训记录
   - **贯穿全栈**

4. **人员资质（WorkerCertification）** — 字段：人员ID、资质类型（特种作业证/安全员证/电工证/焊工证/高空作业证等）、证书编号、发证机构、发证日期、有效期至、证书照片URL、审核状态（Pending/Approved/Rejected/Expired）
   - 业务动作：提交资质、审核资质、到期提醒、续期管理
   - **贯穿全栈**

### 核心页面（≥ 4 个）

1. **安全概览仪表盘** `/dashboard` — 消费：SafetyIncident + SafetyInspection + SafetyTraining
   - 近30天事件趋势折线图、风险等级分布饼图、待整改项列表、本月培训完成率
   - 使用 `<StatusBadge>` 和 `<RiskLevelTag>` 共享组件

2. **事件管理** `/incidents` — 消费：SafetyIncident
   - 事件列表（支持按严重等级、状态、时间筛选）、新增事件表单（含图片上传）、事件详情（时间线展示调查和整改过程）
   - 使用 `<StatusBadge>` 和 `<RiskLevelTag>`

3. **检查管理** `/inspections` — 消费：SafetyInspection
   - 检查计划列表、执行检查（逐项勾选是否合格+拍照）、检查结果报告页
   - 使用 `<StatusBadge>` 和 `<EmptyState>`

4. **培训管理** `/trainings` — 消费：SafetyTraining
   - 培训日历视图、培训详情（签到表+成绩表）、培训记录导出
   - 使用 `<StatusBadge>` 和 `<UserAvatar>`

5. **资质审核** `/certifications` — 消费：WorkerCertification
   - 待审核列表、资质详情（含证书照片预览）、过期预警列表
   - 使用 `<StatusBadge>` 和 `<RiskLevelTag>`

### 横切关注点（≥ 2 个）

1. **RBAC 权限控制**
   - 角色：Admin / SafetyManager / Inspector / Worker
   - 触达文件层：DB role表 + 后端RBAC中间件 + 前端路由守卫 + 前端按钮显隐（Worker只能上报事件不能审核）+ 类型定义 roles.ts

2. **审计日志**
   - 记录所有安全相关操作（事件上报、检查提交、培训签到、资质审核）
   - 触达文件层：DB audit_log表 + 后端日志中间件 + 后端各service埋点 + 前端日志查看页 + 类型定义

3. **全局异常处理**
   - 触达文件层：后端统一异常过滤器 + 后端各controller抛出业务异常 + 前端ErrorBoundary + 前端API拦截器 + 错误码枚举

### 共享枚举/常量（≥ 2 组）

1. **严重等级枚举 SeverityLevel**：NearMiss / Minor / Moderate / Major / Fatal
   - 后端：`backend/src/types/enums.ts`；前端：`frontend/src/types/enums.ts`
   - README 必须列出此枚举在前后端的所有出现位置

2. **事件状态枚举 IncidentStatus**：Reported / Investigating / Resolved / Closed
   - 后端：`backend/src/types/enums.ts`；前端：`frontend/src/types/enums.ts`

3. **检查类型枚举 InspectionType**：Routine / Special / PreShift / Emergency

4. **资质审核状态枚举 CertStatus**：Pending / Approved / Rejected / Expired

### 共享前端组件清单（≥ 3 个）

1. **`<StatusBadge>`** — 消费页面：仪表盘、事件管理、检查管理、培训管理、资质审核
2. **`<RiskLevelTag>`** — 消费页面：仪表盘、事件管理、资质审核（按风险等级着色）
3. **`<UserAvatar>`** — 消费页面：事件详情、培训管理、资质审核
4. **`<EmptyState>`** — 消费页面：所有列表空状态

### 共享 hooks/utils 清单（≥ 2 个）

1. **`useIncident`** — 封装事件详情获取与状态流转逻辑
2. **`usePagination`** — 通用分页逻辑
3. **`getSeverityColor`** — 根据严重等级返回对应颜色常量

### 后端中间件清单（≥ 2 个）

1. **`authMiddleware`** — JWT 鉴权
2. **`rbacMiddleware`** — 角色权限校验
3. **`auditLogMiddleware`** — 审计日志记录
4. **`errorHandlerMiddleware`** — 统一异常捕获
5. **`uploadMiddleware`** — 图片上传处理（Multer）

### 部署要求

项目必须支持 Docker Compose 一键部署，且在任意目录名下正常启动。

需要提供：`docker-compose.yml`、`.env`、`frontend/Dockerfile`、`backend/Dockerfile`、`frontend/nginx.conf`、`.env.example`

docker-compose.yml 要求：
- 不要写 `version:` 字段
- 顶层声明 `name: safety-platform`
- 容器名带 `${COMPOSE_PROJECT_NAME:-safety-platform}` 前缀
- 数据库使用命名卷持久化
- 前端 Nginx 反向代理 `/api` → `http://backend:3000/`
- 数据库 `healthcheck` + 后端 `depends_on.service_healthy`
- 前端 `${FRONTEND_PORT:-80}:80`，后端 `${BACKEND_PORT:-3000}:3000`

自检清单：
1. 中文目录名下 `docker compose config --quiet` 不报错
2. `docker compose ps` 显示全部 healthy
3. 浏览器访问 18702 能看到页面
4. Docker 启动命令是 `docker compose up -d`
